# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

**This file is the fast architectural reference.** For how a specific feature works in detail, see `docs/FEATURES.md`. For shipped/remaining status and the reasoning behind the roadmap, see `docs/ROADMAP.md`.

## Commands

```bash
npm run dev      # build in watch mode (auto-rebuilds on file save)
npm run build    # production build → dist/
npm run zip      # package dist/ into snipkeep.zip for distribution
```

No tests or linter configured. **`npm run build` does NOT type-check** — `vite-plugin-web-extension` transpiles via esbuild, which strips types silently. `tsconfig` has `strict` + `noUnusedLocals`/`noUnusedParameters`, but those only bite if you run the type-checker yourself: use **`./node_modules/.bin/tsc --noEmit`** directly, not `npx tsc` — `npx` has, at least once, resolved to an unrelated registry package literally named `tsc` instead of this project's TypeScript. Known **pre-existing** tsc errors that esbuild ignores (leave them unless fixing that area): `getAuthToken`/`getAuthTokenSilent` (`@types/chrome` now types the callback as `GetAuthTokenResult`, not `string`), `popup.css?inline` (no ambient module decl), and an unused `React` import in `Popup.tsx`.

After every build, reload at `chrome://extensions` → SnipKeep → refresh icon, then refresh the tab you're testing — old content scripts keep running until the tab reloads. **Changing `manifest.json` permissions requires a full reload** (Chrome won't pick them up otherwise — this includes `contextMenus`, `clipboardWrite`, and the host permissions for `web.archive.org`, `api.openai.com`, `api.anthropic.com`, and `generativelanguage.googleapis.com`).

**A note on this Bash environment:** the working directory has drifted mid-session more than once (a command run from `clipnote/` later reports `pwd` as its parent). Chain `cd /path/to/clipnote && pwd && <command>` in one call rather than trusting directory persistence across separate tool calls.

## Architecture

Manifest V3 Chrome extension. Vite + `vite-plugin-web-extension` reads `manifest.json` and compiles every referenced source path into `dist/`. `public/` (icons, bundled font) is copied to `dist/` root. `@/*` aliases `src/*`.

What SnipKeep does: select text on any page → a floating toolbar saves it to a chosen Google Doc; a right-side drawer manages destinations and history. The **Google Doc is the real product** — formatting quality there matters most, and there is no SnipKeep server or database — everything lives in the user's own Doc or on-device. That architectural fact is the whole thesis behind the roadmap in `docs/ROADMAP.md`; don't build a feature that quietly undoes it.

The clip is more than plain text: an optional **margin note** ("your take"), preserved hyperlinks, right-click image capture, a keyboard-driven toolbar, **lecture-timestamp clipping** (clips saved on a YouTube watch page carry `videoTime` — the transcript line's own DOM timestamp when the selection sits in one, else playback time; the Doc bullet gets a caption-styled " · 43:21" link and History's Source link reopens the video at that moment. The timestamp is NEVER baked into `sourceUrl`, which doubles as page identity for article-grouping/Works-Cited-dedup/archiving — see `src/lib/video.ts`), full-text-searchable history with citations (plus an auto-maintained **Works Cited** section in the Doc), an optional **bring-your-own-AI-key** layer (follow-up questions per clip, per-doc summaries — user's own OpenAI/Anthropic/Gemini key, straight from the browser, consistent with the no-server thesis), and behavioral-psychology-informed features (deadlines, reflection nudges, resurfacing) — see `docs/ROADMAP.md` for the full list and `docs/FEATURES.md` for mechanics. (Soft Triage/Someday was built, then fully removed 2026-07-09 on user request.)

> Naming: the product was renamed **ClipNote → SnipKeep**. The wordmark is rendered as two colored spans — `Snip` (white) + `Keep` (accent gold) — in `Drawer.tsx` and `Popup.tsx`, so it never exists as the joined string "SnipKeep" in JSX. Internal ids use the `snipkeep-` prefix; the signing key is `snipkeep-key.pem` (gitignored; the extension id comes from the `key` field in `manifest.json`, which is unchanged).

### Three isolated execution contexts

**Background** (`src/background/index.ts`) — MV3 service worker. All API calls (Google Docs; the three BYO-AI providers; Notion code exists but is hidden at MVP), OAuth via `chrome.identity`, history, per-destination tracking, and **all `chrome.identity` work** (see gotcha below). Can be killed/restarted anytime — never rely on in-memory state; persist to `chrome.storage`.

**Content script** (`src/content/index.tsx`) — injected into every page except `docs.google.com/document/*` (the actual Docs editor — every SnipKeep destination is a `/document/d/.../edit` URL, so this is specifically "don't clip from the doc you're clipping *into*"; `manifest.json`'s `exclude_matches`, no in-code hostname check backs it up). This used to blanket-exclude all of `docs.google.com`, which also silently killed the toolbar on Drive file previews (`docs.google.com/file/d/.../preview`) — a different Google product surface (DOM-based PDF/image viewer) hosted on the same domain, not the Docs editor, with no reason to be excluded. Narrowed on request. Guarded against double-injection via `window.__snipkeepLoaded`. Renders the floating **Toolbar** on `mouseup` (250ms debounce), opens/closes the **Drawer** on `TOGGLE_DRAWER`, saves directly on `TRIGGER_SAVE` (Cmd/Ctrl+Shift+S) with a toast, drives the toolbar from page-level keys, extracts preserved links (`extractLinkSpans`), and handles `CAPTURE_IMAGE`.

**Gotcha — `all_frames: true` + `chrome.tabs.sendMessage` without `frameId` means "broadcast to every frame," not "send to the top frame."** The content script now runs in every frame (`manifest.json`, `all_frames: true`) so the toolbar's `mouseup` detection reaches into same-page iframes — needed for pages like Google Drive's file-preview viewer, which renders its actual selectable content inside a nested iframe, invisible to a top-frame-only content script. But `chrome.tabs.sendMessage(tabId, msg)` with no `frameId` option delivers to **every frame's** listener, not just frame 0 — confirmed against Chrome's own docs, not assumed. Each frame is a fully separate JS context (its own module-level state), so an unguarded broadcast handler would run redundantly, or worse, duplicate the drawer into an iframe. Three messages cross this boundary, each fixed differently based on what it actually needs:
- `TOGGLE_DRAWER` and `CAPTURE_IMAGE` are sent from the background with an explicit `frameId` (`0` for the drawer — always page-level, never per-frame; `info.frameId` from the context-menu click for images — the exact frame the right-click happened in), so they're never delivered anywhere else in the first place.
- `TRIGGER_SAVE` (the keyboard shortcut) can't be targeted this way — `chrome.commands` gives the background no frame information — so it's still broadcast to every frame, and the content-script handler itself opens with `if (!document.hasFocus()) return`. Only the frame the user was last interacting with (where a selection would actually live) reports focus; every sibling frame silently no-ops instead of showing its own spurious "no text selected" toast.

Any *new* message sent via `chrome.tabs.sendMessage` needs the same thought: either target an explicit `frameId`, or guard the handler so only the relevant frame acts on it.

**Known, confirmed-unfixable limitation — the toolbar cannot appear over PDF content, anywhere, in any Chrome tab.** Not specific to Google Drive, not fixable via `all_frames`/`exclude_matches`/any manifest or content-script change. Investigated in depth (live DOM inspection of `docs.google.com/file/d/.../preview` and `/view`: no iframe large enough to hold the content, no canvas, no matching text nodes, no Shadow DOM, `window.getSelection()` empty immediately after a real drag-selection) before concluding this. The resolving fact: **any** raw `.pdf` URL opened directly in Chrome — no Google Drive involved at all — silently redirects to `chrome-extension://efaidnbmnnnibpcajpcglclefindmkaj/...`, Chrome's own built-in PDF Viewer, which Chromium documents as walled off from content-script injection (the same restriction as `chrome://` pages). Confirmed three independent ways: (1) Chromium's own documented behavior, (2) browser-automation tooling itself refusing to send input/execute JS on that URL ("browser-internal"), (3) the exact symptom pattern on Google's Drive preview (real, working native copy/paste, but zero DOM footprint anywhere) that this architecture predicts. Google Drive's file/PDF preview almost certainly embeds this same privileged renderer internally, which is why it behaves identically. **Don't re-attempt this** without new evidence Chrome's own PDF-viewer restriction has changed — regular web pages, Sheets, Slides, Forms, and any real DOM-rendered iframe are unaffected and already work correctly.

**Popup UI** (`src/popup/`) — React app (`Popup.tsx` + `popup.css`). Rendered primarily *inside the drawer*; `popup.html` is kept as a build entry but `default_popup` is removed so the icon fires `chrome.action.onClicked`. **Three tabs: Docs** (`DocsTab` — active destinations sorted above inactive ones, divided by a "Hidden from toolbar" label; "Add document" sits right after the active group, not the true bottom of the list), **Completed** (`CompletedTab` — finished projects), and **History** (the clip archive — search, a document Filter, per-clip Cite/menu actions, single or bulk delete with undo; ✨ Resurfaced exists in code but is currently paused, see `docs/ROADMAP.md`). See `docs/FEATURES.md` for the mechanics of each — this file only tracks the patterns/gotchas below that apply across multiple features.

### `chrome.identity` is NOT available in content scripts (the #1 gotcha)

The drawer (and the Popup it renders) runs in the **content-script** context, where `chrome.identity` is `undefined`. Any identity/auth/Docs-token work from the drawer must be routed to the background via `chrome.runtime.sendMessage`. Handlers:
- `GET_USER_PROFILE` → avatar email + display name (`getProfileUserInfo` for email; name via Google's `oauth2/v3/userinfo` endpoint, requiring the `userinfo.profile` OAuth scope — see below)
- `GET_DOC_TITLE` → resolve a Doc's title (silent token + Docs API)
- `SIGN_IN` (gate screen) → interactive OAuth + cache email/name
- `SIGN_OUT` (drawer) → `removeCachedAuthToken` + clear `isSignedIn`/`userEmail`/`userName`

Background auth helpers: `getAuthToken()` (interactive) and `getAuthTokenSilent()` (non-interactive, never prompts). Sign-out: `Drawer.handleSignOut` sends `SIGN_OUT` and optimistically clears the avatar; the `chrome.storage` change flips `isSignedIn` → the gate screen. (`SIGN_OUT` only drops Chrome's cached token, not the Google-side grant, so re-sign-in is a fast silent re-grant.)

**Display name gotcha:** `chrome.identity.getProfileUserInfo()` only ever returns `{email, id}` — never a name, regardless of scopes granted; that's a hard limit of the Chrome-level API, not a config issue. The only way to get a real name is Google's own `oauth2/v3/userinfo` endpoint via `fetch()` with the OAuth token, which requires the `userinfo.profile` scope (added to `manifest.json` alongside the existing `documents` scope). Adding a new scope means **already-signed-in users need to sign in again** before a silently-fetched token can include it — `getUserName()` in the background handles this by failing silently (returns `null`) rather than erroring, and the UI (`Drawer.tsx`) already treats a missing name as "not available yet, show email only," never a broken state.

The avatar dropdown also has a **🔒 Privacy** entry and, on first use, a **Trust Card** — see `docs/FEATURES.md` for both. The drawer header additionally carries an always-visible **✨ AI** button (the BYO-AI showcase entry point — deliberately NOT duplicated in the avatar dropdown; one path in). The drawer body is a small view state machine: `view: 'main' | 'privacy' | 'trust' | 'ai'` in `Drawer.tsx` — nested screens, not extra tabs, per the Hick's-Law precedent set when Completed became a tab.

### Drawer (Radix Sheet in Shadow DOM)

`src/content/Drawer.tsx` mounts the Popup as a right-side sheet built on `@radix-ui/react-dialog` (wrapped in `src/components/ui/sheet.tsx`). Key adaptations:
- **`modal={false}`** + `onInteractOutside` prevented + no overlay — SnipKeep is a *companion*: the page must stay scrollable/interactive, and the drawer stays open until closed via ✕ / Esc / icon toggle.
- Radix portals to `document.body` by default; `SheetContent` takes a `container` prop so the portal stays inside the Shadow DOM.
- Closing runs one path (`close()` in Drawer): set `open=false` to play the slide-out, then unmount after ~240ms. `closeRef` lets the content script trigger it.
- **Three `<style>` tags injected in order**: `POPUP_CSS` (popup.css `?inline`, `:root`→`:host`, `body`→`.cn-popup-root`), then `SHEET_CSS`, then `BODY_CSS` (drawer chrome).

### Fonts (bundled, not Google Fonts)

Plus Jakarta Sans is bundled at `public/fonts/plus-jakarta-sans.woff2` (one variable file, 400–800). External `@import` is blocked by strict page CSPs inside content scripts, so it's exposed via `web_accessible_resources` and loaded by `ensureFontLoaded()` (`src/lib/fonts.ts`), which injects the `@font-face` into **`document.head`** (NOT a shadow root — Chrome doesn't reliably apply shadow-scoped `@font-face`). Called from `content/index.tsx`, `Drawer.tsx`, and `popup/main.tsx`.

### Icon click → drawer (with inject-on-demand)

`chrome.action.onClicked` messages the tab with `TOGGLE_DRAWER`. Tabs opened *before* the extension loaded have no content script, so on failure the background injects `src/content/index.js` via `chrome.scripting.executeScript` and retries. On restricted pages (New Tab, `chrome://`, Web Store, PDF viewer) injection is impossible → a `chrome.notifications` message explains why. The right-click "Save image" context menu uses the same inject-on-demand fallback.

### Voice tab (`src/voice/`) — voice-note capture

The Toolbar's mic button (speak a margin note instead of typing it) runs `SpeechRecognition` inside a **real, visible tab** (`chrome.tabs.create({url: 'src/voice/index.html'})`), not the content-script context and — this took two attempts to land on — not a `chrome.offscreen` document either.

**Why not a content script:** a mic permission requested there is scoped to *whatever website the user is on* ("nytimes.com wants to use your microphone") — asked again per new domain, silently blocked outright on sites with a restrictive Permissions-Policy header.

**Why not `chrome.offscreen` (tried first, reverted):** offscreen documents have no visible surface for Chrome to anchor a `getUserMedia` prompt to, so `recognition.start()` there fails immediately with `NotAllowedError`, no dialog ever shown — confirmed against the Chromium extensions mailing list and multiple GitHub issues describing the identical failure. The natural-seeming fix — grant the permission via a separate real tab first, then let the offscreen doc (same extension origin) use it afterward — was built and tested live, and **still failed**: granting via the tab worked (confirmed: native prompt appeared, "Microphone in use" indicator lit up), but the offscreen document still couldn't use the mic afterward. Since no source could be found confirming offscreen documents can *ever* successfully call `getUserMedia`, even post-grant, the whole approach was abandoned rather than guessed at a third time. `src/offscreen/` and `src/permission/` (the two-file split from that attempt) no longer exist.

**Current design:** one real tab does both jobs — requesting the permission (shows Chrome's native dialog on first use; resolves instantly with no prompt on every use after, since the grant is scoped to the extension's stable origin) and running the recognition itself, live, for as long as the tab stays open. `vite.config.ts` needs `additionalInputs: ['src/voice/index.html']` — like the abandoned offscreen/permission pages, this isn't declared anywhere in the manifest schema (`chrome.tabs.create()` just takes a URL string at runtime), so without this the plugin never bundles it into `dist/`.

**The tab opens in the background (`active: false`), not focused.** The user should watch the note field fill in live on the page they're clipping from, not have their view yanked to a separate tab — a real, visible tab is required for the mic to work at all (see above), but "visible" doesn't have to mean "focused the whole time." The voice tab decides *before* calling `getUserMedia` whether it actually needs the user's attention: it checks `navigator.permissions.query({name: 'microphone'})` first, and only sends `VOICE_TAB_NEEDS_FOREGROUND` (background responds with `chrome.tabs.update(tabId, {active: true})`) if the state isn't already `'granted'` — i.e., the very first grant, or explaining a past denial. Every use after that first grant, the tab does its job silently in the background and the user never needs to look at it. This check has to happen *ahead* of calling `getUserMedia`, not in reaction to it — a backgrounded tab may not even be able to show Chrome's native dialog once it's already trying to appear, so the decision to foreground has to be made first.

```
Toolbar --START_VOICE_NOTE--> background --(chrome.tabs.create, active:false)--> voice tab
voice tab --VOICE_TAB_NEEDS_FOREGROUND--> background --(chrome.tabs.update, active:true)--> voice tab [only if mic permission isn't already granted]
voice tab --VOICE_RECOGNITION_EVENT--> background --VOICE_NOTE_UPDATE (explicit frameId)--> Toolbar
Toolbar --STOP_VOICE_NOTE--> background --VOICE_TAB_STOP (explicit tabId, chrome.tabs.sendMessage)--> voice tab
```
The background tracks `voiceSession: {originTabId, originFrameId, voiceTabId}` in a plain in-memory variable — fine for a live-streaming interaction, unlike the durable data elsewhere in this file that must survive a service-worker restart; if the SW is killed mid-recording, that one session just stops. Because the voice tab is a real tab (not an offscreen doc), `chrome.tabs.sendMessage` can target it directly by id for the stop command — no broadcast-collision risk to design around on that leg, unlike the origin-tab relay leg which still needs an explicit `frameId`. A `chrome.tabs.onRemoved` listener also ends the session if the voice tab is closed any other way (user closes it manually, or it self-closes after an error), so state can't linger stale. Ending a session always calls `chrome.tabs.update(originTabId, {active: true})` to explicitly return focus to wherever the user was working — `chrome.tabs.create`'s own opener-based focus-return isn't reliable when the tab was opened from the background rather than a real click in the origin tab.

**The finishing interaction is deliberately psychology-first, not just "whatever was easiest to build."** Auto-stop is silence-triggered, not a second click — a longer `INITIAL_SILENCE_MS` (8s) gives someone time to start talking, a much shorter `PAUSE_SILENCE_MS` (1.8s) is what actually fires once they've spoken and gone quiet, matching the pause-means-done model every voice assistant already trains people on. Auto-stopping the *listening* is deliberately not the same as auto-*saving* — the transcript just sits in the note field, editable, until a real Enter/Save action, preserving one review beat before anything is written to the Doc. Enter (or any of the toolbar's save-triggering controls — `handleSaveRequest` covers all of them) while still recording means "finish and save," not "do nothing": it stops recognition and defers the actual save until the real `'ended'` event confirms the final transcript landed, via a ref-based `handleSaveRef` (not called directly from the message listener, which registers once and would otherwise close over a stale `note`). See `docs/FEATURES.md`'s "the finishing interaction" entry for the full reasoning, including a real edit-clobbering bug this design pass caught (a manual fix to a mis-heard word mid-recording used to get silently overwritten by the next transcript update) and how it's fixed (`noteValueRef`/`lastVoiceWriteRef` detect a manual edit and switch to appending rather than replacing).

**Gotcha — the deferred save can race the silence auto-stop, since both can be triggered by the exact same moment.** Pressing Enter mid-speech and the silence-timer's own auto-stop are both reacting to "the user just stopped talking," so they can land within milliseconds of each other. If the background has already cleared `voiceSession` (auto-stop's `'ended'` already processed) before the Toolbar's `isRecording` state catches up, `handleSaveRequest` reads a stale `true`, sends a second `STOP_VOICE_NOTE` that's now a no-op (nothing left to stop), and there's no second `'ended'` event coming to ever resolve `saveAfterStopRef` — the save would wait forever. Fixed with a 1.2s safety-timeout alongside the deferred-save flag: if nothing has cleared it by then, save anyway with whatever's currently in `note`. Not an attempt to perfectly synchronize two independently-timed async triggers — the real state only ever exists behind message-passing, never synchronously — just a guarantee the save always eventually happens regardless of how the race actually settles.

### Message flow

```
Icon click   → background onClicked      → TOGGLE_DRAWER (+ inject-on-demand fallback) → content → Drawer
Cmd+Shift+S  → background commands       → TRIGGER_SAVE  → content (direct save + toast)
Save text    → content → SAVE_NOTE        → background → Google Docs API   (text, note, links)
Save image   → background contextMenus.onClicked → CAPTURE_IMAGE → content (read img size/title)
             → content → SAVE_IMAGE       → background → Google Docs API   (insertInlineImage)
Add doc note → content → ADD_DOC_NOTE     → background → Google Docs API   (Living Resurface write-back)
Works Cited  → content → UPDATE_BIBLIOGRAPHY → background → Google Docs API (delete old block, rebuild at doc end)
AI connect   → content → CONNECT_AI / DISCONNECT_AI → background (validate key via provider's models endpoint, store/remove aiConfig)
AI actions   → content → ASK_FOLLOWUP / SUMMARIZE_TOPIC → background → user's own AI provider (callAI adapter)
Drawer auth  → content → GET_USER_PROFILE / GET_DOC_TITLE / SIGN_IN / SIGN_OUT → background (chrome.identity)
Voice note   → content → START_VOICE_NOTE → background → chrome.tabs.create → voice tab (src/voice/)
             → voice tab → VOICE_RECOGNITION_EVENT → background → VOICE_NOTE_UPDATE (explicit frameId) → content
             → content → STOP_VOICE_NOTE → background → VOICE_TAB_STOP (explicit tabId) → voice tab
```
All message types are in `src/types.ts`.

### Bring-your-own-AI-key layer (`callAI` in the background)

The user connects their own OpenAI/Anthropic/Gemini key from the drawer's ✨ AI screen; SnipKeep never sees or relays it (no-server thesis intact). Everything AI goes through one `callAI(systemPrompt, userPrompt)` adapter in the background, dispatching on `aiConfig.provider`. Raw `fetch`, **not** the provider SDKs — this codebase calls every external API (Docs, Notion, archive.org) via plain fetch from the MV3 worker; don't introduce an SDK for one provider. Models are fast/cheap tiers on purpose (utility calls, not heavy reasoning): `gpt-5-nano` / `claude-haiku-4-5` / `gemini-flash-latest` (a durable Google alias, chosen over a dated snapshot so retirement doesn't silently break it).

Three gotchas, all found via live testing:
- **Anthropic rejects browser-origin requests by default** — every Anthropic fetch needs the `anthropic-dangerous-direct-browser-access: true` header (the documented opt-in for exactly this architecture; same flag the official SDK sets under `dangerouslyAllowBrowser`). Host permissions bypass the *browser's* CORS check, not Anthropic's own server-side one — both layers are needed.
- **Read the provider's error body, don't guess from the status code.** All three providers shape errors as `{ error: { message } }`; `aiErrorMessage()` surfaces that verbatim. A 401/403 covers several distinct causes (bad key, no billing, model access) that each need a different user fix — a generic "key rejected" message actively misled during testing (the real error was the CORS header above).
- **Claude Pro ≠ API access.** A consumer subscription carries zero API credit; the console needs its own billing/credits. Expect users to hit this — the surfaced error text is the diagnostic.

Keys are validated at connect time with a free models-list GET per provider (fails fast, no completion cost). UI rule: when no key is connected, AI menu items are `undefined`/omitted entirely — never rendered disabled. Prompting rule (from the PACER critique in `docs/IDEAS.md`): AI asks questions and classifies; it never writes the reflection/analysis *for* the student.

### Works Cited write-back (delete-then-rebuild, not append)

Citing a clip (any cite, first or re-cite) rebuilds a "Works Cited" block at the true end of that Doc: `updateBibliography()` resolves the block's NamedRange (`worksCitedBookmarks`), **deletes the whole old block**, re-fetches the doc end, inserts the fresh list, and stores the new NamedRange id. Rebuild-not-append is the point — it's the only way the list stays deduplicated (one entry per source *page*, not per clip), alphabetized, at the actual bottom, and in ONE consistent style (the whole list re-renders in whichever style was just picked; per-entry historical styles would mix). `resolveNamedRange` returns `{ startIndex, endIndex }` — startIndex exists specifically for this delete; the older callers (archive-link, add-doc-note) only read `endIndex`. The UI treats the Doc write as secondary to the clipboard copy: silent on success ("Copied ✓" already confirmed the action), but failure surfaces as "Doc update failed" in the same button-flash slot — and `.history-item.show-actions` pins the actions row open while any feedback shows, because the Cite dropdown closing moves the cursor onto the next card and would otherwise collapse the row before the feedback is readable. Known limits (documented, not hidden): the block only repositions on a cite (clips saved after it stay below it until the next cite), and Google-Docs-only like every other write-back.

### Design system / visual hierarchy

Tokens live in `popup.css` `:root` — the **"Ink & Highlighter" palette** (2026-07-14, replacing the original violet after an OKLCH/APCA-checked three-way study): warm ink-black surfaces (even OKLCH lightness steps L 0.16→0.29 at hue ~75°) + one marker-yellow accent (`--accent #F6E12B` — semantically motivated: the product IS a highlighter) + semantic state colors `--danger` (red) and `--warn` (**orange `#E78A45`, deliberately re-hued away from amber** so deadline chips can't be mistaken for the gold accent). Two accent-tint helpers: `--accent-dim` (0.10 — focus-ring glow / flash fill) and `--accent-soft` (0.18 — **the** "active pill" fill; every selected chip/toggle/cite-option must use it, never a hand-rolled `rgba(246,225,43,…)`, or they drift). The accent-soft alpha ceiling is 0.34 for marker yellow (4.63:1; 0.18 gives 7.45:1) — recomputed for this accent, don't raise past 0.34 without re-running the check. Text ramp `--text` / `--text-2` / `--text-3` is **WCAG-locked** (don't dim `--text-3` below `#979189` — 5.2:1 on cards). The margin-note color written into the actual Google Doc (`NOTE_FG` in `background/index.ts`) is part of this palette too: dark marker-yellow ink `#71631D`, 6.0:1 on the Doc's white page. Hierarchy is carried by a 4-tier type scale + weight + color together; accent reserved for ~4 tiny spots. Avoid one-off font-size bumps — they flatten the scale.

**Spacing:** a 4px-grid scale lives in `:root` as `--space-1`…`--space-6` (4/8/12/16/20/24). Gaps, padding, and margins should reference these, not raw px. Do **not** stack a child `margin-bottom` on top of a parent flex `gap` — that double-spaces (History's control cluster hit exactly this: `gap:22` + per-child margins → 32–34px real gaps). Group related controls under one wrapper with its own `gap` (proximity) instead. Content cards (`.doc-item`/`.history-item`/`.account-row`/`.privacy-item`) all share `var(--space-3) var(--space-4)` padding.

**Tab switching:** instant, no entrance animation — a `.tab-content` mount animation was tried and then deliberately removed (see `docs/FEATURES.md`). The only motion left in the popup is the tuned card-reorder slide in `DocsTab`; don't reintroduce a whole-section fade/settle on tab switch without being asked.

**Gotcha:** `Toolbar.tsx` and the toast (`content/index.tsx`) are separate Shadow DOM roots that **hardcode hex values** (no access to the CSS variables). When tokens change, update those by hand or they silently drift out of WCAG sync.

**Icons:** `react-icons/md` (Google's Material Design set) is the icon system across `Popup.tsx`, `Drawer.tsx`, and `Toolbar.tsx` — replaced an earlier mix of emoji (🔒✨🕒📄🏛💭📋, inconsistent across OS/platform, colorful against a monochrome UI) and plain Unicode glyphs. Chosen over bundling Google's actual Material Symbols variable font (which would ship every icon in one large file just to use ~20) — `react-icons` tree-shakes per-icon like `lucide-react` already does elsewhere in this codebase; the entire sweep (~25 call sites, 17 distinct icons) added **~8.7KB** to the bundle. Icons render with `fill="currentColor"`, so they inherit each button's existing text color automatically — no separate color wiring needed, including in Toolbar's hardcoded-hex context. `lucide-react` is still used for two things deliberately left alone: `Bookmark` (Drawer's logo mark — brand identity, not a utility icon) and `sheet.tsx`'s `XIcon` (dead code — `Drawer.tsx` passes `showClose={false}`, so it never renders; SnipKeep has its own close button already).

### Card overflow menus (`.card-menu`) — shared pattern, one real gotcha

Every card-level "···" popover (`DocMenu`, `HistoryCardMenu`, `CiteMenu`, `FilterMenu`) shares one `.card-menu` CSS surface and one behavioral pattern: dismiss-on-outside-click via `e.composedPath()` + a ref, rendered as a `position: absolute` sibling anchored to a `position: relative` wrapper. New popovers should reuse this pattern rather than invent a new one.

**Gotcha — opacity on the anchoring card can silently swallow its own popover.** Any element with `opacity < 1` (e.g. `.doc-item.inactive`, `.doc-item.done`) establishes its **own CSS stacking context**. If that card's popover is open, the popover — regardless of its own `z-index` — is trapped inside the dimmed card's stacking slot, and a *later* sibling that's also opacity-dimmed can still paint over the whole thing (sibling order is resolved by the *cards'* stacking position, not the popover's). Real bug, found via screenshot, fixed by promoting the anchoring card itself while its popover is open (`.doc-item.popover-active { position: relative; z-index: 30; }`, driven by `menuOpenFor === doc.id || isEditingDeadline` in the class list) — not by raising the popover's own z-index, which doesn't escape the trap. See `docs/FEATURES.md` for the full mechanism. Any new dimmed/opacity-modified element that gets its own popover later needs the same treatment.

**Gotcha — a `<button>` can never contain another `<button>` (the avatar dropdown had this bug for a long time).** The account dropdown's trigger used to literally wrap its own dropdown, Privacy button and Sign-out button included, inside `<button className="cn-avatar">`. That's invalid HTML — interactive content can't nest — so the browser's parser silently force-closes the outer `<button>` the instant it hits the first nested one, rewriting the actual DOM to something that doesn't match the JSX at all (confirmed by diffing `--dump-dom` output against the source). Privacy/Sign-out ended up as siblings of the avatar button in the real tree, outside the dropdown's own positioned box — invisible as a bug until the dropdown's content changed enough to make the layout break visible. Fixed: the trigger button and its dropdown are now siblings under one `position: relative` wrapper (`.cn-avatar-wrap`, holding the ref used for outside-click detection), never parent/child. Any future dropdown/menu whose trigger is a `<button>` must follow this shape — never put other interactive elements inside the trigger itself, even conditionally.

### List reorder animation (FLIP) — use the Web Animations API, not CSS transitions

`DocsTab`'s active/inactive sort animates via FLIP (First-Last-Invert-Play): a `useLayoutEffect` measures each participant's `offsetTop` every render and, when it changed, plays the delta as a slide via **`el.animate()`** (native Web Animations API) — deliberately *not* `el.style.transition`/`el.style.transform`. Two real bugs came from the CSS-transition version before it was replaced: (1) setting `el.style.transition` **replaces** the card's own CSS transition for its hover highlight, so the highlight snapped instead of fading for the whole slide; (2) manual reflow-forcing + `requestAnimationFrame` release timing is fragile. `.animate()` avoids both — it runs as an independent animation layer that can't conflict with CSS transitions, and the browser applies the first keyframe deterministically with no timing tricks.

The choreography is "pause, then move": a flat `REORDER_HOLD_MS` (140ms) hold before every slide, plus a small per-card stagger. The divider's CSS `animation-delay` (0.14s) is **kept in sync with it by hand** — grep for `KEEP IN SYNC` before changing either. These invariants were each a real, user-reported bug; re-read this list before touching the animation:

- **`fill: 'backwards'`, never `'none'`, on any delayed WAAPI animation.** Keyframes only apply during the ACTIVE phase — with `'none'`, the element sits at its final layout position for the whole `delay`, then snaps BACK to the first keyframe when the active phase starts. A delayed animation with `fill:'none'` is "jump to end → hold → jump back → slide": worse than no delay.
- **Interruption continuity.** Rapid re-toggling must not restart from the stale layout delta — `runningAnims` (our own per-key `Animation` map, deliberately NOT `el.getAnimations()`, which would also cancel the card's CSS hover transitions) lets the new slide fold the in-flight transform/opacity (read from computed style) into its start point, cancel the old animation, and skip the hold (frozen-mid-air reads as a hang). `interrupted` is its own animate trigger — a quick toggle-back can land on dy≈0 while visibly displaced.
- **Guarded cleanup.** Animation `cancel` events fire async — an old animation's cleanup must check it's still the animation on record before clearing `will-change`, or it strips layer promotion out from under its replacement.
- **Everything that visibly moves participates.** The "Add document" control is anchored after the last active card and moves on toggles; it joins the FLIP pass under a sentinel key (`__add-control`). It REMOUNTS as a new node when it moves (different card's Fragment), which FLIP tolerates (history keyed by sentinel, not element) — but the in-flight fold-in must verify `prevAnim.effect.target === el` first.
- **Data arriving is not movement.** Docs load async from storage, so render 1 is always empty and render 2 populated; anything existing in both renders (the add control) would ghost-slide on every drawer open. Guard: if no CARD has position history, record positions, animate nothing (`isInitialPopulation` in the effect; `dividerMayAnimate` is the render-time mirror gating the divider's `.entering` CSS mount animation — CSS can't know WHY an element mounted, so React decides).
- **`will-change: transform, opacity`** wraps each slide (set before `animate()`, cleared on finish/cancel): the mouse is necessarily hovering the clicked card, so its `box-shadow` hover transition (main-thread repaint) starts the same frame as the slide; layer promotion isolates them.
- **Hover-reveals share the same rhythm.** History cards' action-row expansion uses the same curve and a 0.14s `transition-delay` hover-intent hold (grazed cards never move — a delayed CSS transition cancels outright if the value reverts first), with a 0.1s forgiveness delay on collapse. Any property a `:hover` rule changes must be in the element's `transition` list — `border-color` and `opacity` were both once missing, each snapping at the exact onset of otherwise-smooth motion.
- **Insertions animate as height growth, not FLIP.** A conditionally-mounted element can't animate open (CSS needs a "before" state), so opening the add-form used to pop it in at full height while the divider (not a FLIP participant) and cards below teleported/held on different clocks. The add control's trigger + form are now permanently co-mounted in reciprocal 0fr↔1fr grid collapses (`.add-collapse`) — one smooth net height change that everything below rides, both directions. FLIP stays out of it: during a height transition the effect measures pre-transition layout, sees dy≈0, and correctly does nothing. (`autoFocus` dies with permanent mounting — focus explicitly on open.)
- **Two motion registers, don't mix them.** The slow symmetric curve + hold is for SYSTEM-initiated rearrangement (reorders). Direct user commands (opening the add-form) use a fast ease-out (0.22s, `cubic-bezier(0.16, 1, 0.3, 1)`) — respond immediately, no hold, no lingering. A user-invoked control animated on the reorder clock reads as laggy, not gentle (user feedback, 2026-07-10).
- **The 0fr-grid collapse trick can't collapse padding.** With the global `border-box`, an element's height never shrinks below its own padding — at-rest padding on the clipped inner leaves an invisible band that reads as lopsided card padding (real user report: "bottom padding is bigger than top"). Padding on `.history-actions-inner` is 0 at rest and applied only in the expanded state, transitioned on the same clock as the row.

**Gotcha — don't trust an easing curve's tangent at a glance; verify the curve numerically.** A curve with a mathematically-zero velocity at t=0 is not necessarily "gentle" in practice — `cubic-bezier(0.4, 0, 0.2, 1)` (Material's "standard" curve) has a zero tangent at the start but still reaches ~24% progress by 25% of the duration, close to linear, not gentle. Verified by solving the actual bezier function (see `docs/FEATURES.md`'s "third pass" note for the numbers) before landing on `cubic-bezier(0.65, 0, 0.35, 1)`, which is genuinely slow at both ends (~7% progress by 25% of duration) and is now the shared curve for the reorder, the history-card expansion, and the divider entrance. If retuning any of these, compute the curve's progress-over-time numerically rather than reasoning from the control points alone.

### Reversible-delete pattern (undo bar)

History's single-clip "Remove from history" and the bulk "Clear" control both go through one shared mechanism: `commitClips(next)` writes state + `chrome.storage.local.clips` together, and `stageUndo(snapshot, label)` keeps the pre-delete array for 6s behind a sticky `.undo-bar` at the list's bottom, restorable via one `applyUndo()`. This is the established pattern for any future reversible destructive action in this popup — prefer it over a blocking confirm dialog for anything frequent/low-stakes (reserve inline confirms, like the Doc card's Remove or bulk Clear, for actions with real stakes or an unclear blast radius).

### Shadow DOM notes

Toolbar (`#snipkeep-host`) and drawer (`#snipkeep-drawer-host`) are Shadow roots. Use `e.composedPath()` (not `e.target`) to detect clicks from the outer document; CSS vars on `:host` are visible inside; host has `pointer-events: none` and the toolbar overflows it.
