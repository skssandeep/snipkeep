# SnipKeep тАФ Feature Deep-Dives

Detailed mechanics for the features built from `research/SnipKeep-Competitive-Psychology-Report.pdf`'s 11-feature roadmap, plus the two pre-roadmap capture features (margin notes, keyboard toolbar) that predate the report. Kept separate from `CLAUDE.md` so that file stays a fast architectural reference тАФ come here when you're touching one of these specific features and need the "why," not just the "what."

See `docs/ROADMAP.md` for shipped/remaining status and priority order.

## Privacy Ledger (report #1)

The avatar dropdown has a **ЁЯФТ Privacy** entry that switches `Drawer`'s `view` state (`'main' | 'privacy' | 'trust'`, mutually exclusive body content) to `<PrivacyLedger onBack={...} onShowTrust={...}/>` тАФ a plain-language, honest restatement of the real architecture (no SnipKeep server; archive is device-local; settings sync via Chrome's own sync, not ours). It's a settings-like destination, not a tab, since it isn't a browse-my-clips action.

### Account dropdown redesign тАФ added display name, restructured for hierarchy (later session)

The avatar dropdown originally showed only email, flat, above Privacy/Sign out. Two changes:

**Added a real display name.** `chrome.identity.getProfileUserInfo()` only ever returns `{email, id}` тАФ never a name, no matter what scopes are granted, a hard limit of that specific Chrome API. Getting a real name requires a *separate* call to Google's own `oauth2/v3/userinfo` endpoint (via `fetch()` with the OAuth token), which needs the `userinfo.profile` scope тАФ added to `manifest.json` alongside the existing `documents` scope. This is a genuine permissions change, not just styling, and was called out to the user as such (new scope тЖТ fresh consent screen on next sign-in) rather than added silently.

The message that used to be `GET_USER_EMAIL` is now `GET_USER_PROFILE`, returning `{ email, name }`; `SIGN_IN`'s response gained an optional `name` field too. New background helper `getUserName(token)` calls the userinfo endpoint and returns `null` on any failure тАФ deliberately silent, not thrown, because **an already-signed-in user's cached token predates the new scope** and can't produce a name until they next sign in interactively. The UI never treats a missing name as broken: `Drawer.tsx` conditionally renders the name line only when present, falling back to email alone. `chrome.storage.sync` gained a parallel `userName` key alongside `userEmail`, cleared together on sign-out.

**Restructured the dropdown into three tiers** (identity тЖТ Privacy тЖТ Sign out), mirroring the shape of Google's own account menu (Jakob's Law тАФ no new pattern to learn):
- **Identity block** (name + email) тАФ name is primary (13px bold, `--text`), email is secondary (11.5px, `--text-3`) sitting just 2px below it тАФ tight, deliberate proximity, since they're one fact (who you are), not two.
- A `border-bottom` closes the identity block, matching the existing full-width-border convention this specific dropdown already used elsewhere (not the inset-margin divider style used in `.card-menu` тАФ different component, its own established look, kept consistent with itself rather than importing a second divider convention).
- All three tiers (identity / Privacy / Sign out) share one padding rhythm, `var(--space-3) var(--space-4)` тАФ the same convention already used for `.doc-item`/`.history-item`/etc тАФ so the menu reads as one consistent rhythm rather than three separately-tuned rows.
- The avatar's own initial letter (`initial` in `Drawer.tsx`) now prefers the **name's** first letter over the email's, falling back to email only when no name is available yet тАФ matches what people actually expect their own avatar to show.

### Account dropdown revamp #2 тАФ a real structural bug, plus a proper profile-card layout (still later session)

The user asked to "revamp this overall profile card" after screenshotting the identity block rendering **centered**, unlike Privacy/Sign out below it (left-aligned). Two things came out of investigating that, one cosmetic and one a genuine, previously-invisible bug:

**The centering was a real CSS-inheritance bug, not a styling choice.** `text-align` is an inherited property, and `<button>` elements carry `text-align: center` in every browser's default stylesheet. `.cn-auth-identity`/`.cn-auth-name`/`.cn-auth-email` never set their own `text-align`, so they inherited *centered* from their ancestor тАФ which, at the time, was `<button className="cn-avatar">`. `.cn-auth-privacy`/`.cn-auth-signout` had always explicitly set `text-align: left`, which is exactly why only the identity block was affected. Fixed by giving `.cn-auth-identity` an explicit `text-align: left` rather than relying on not having an ancestor that overrides it.

**The deeper bug this surfaced: `.cn-avatar` (a `<button>`) had `.cn-auth-privacy` and `.cn-auth-signout` (also `<button>`s) nested inside it.** That's invalid HTML тАФ a `<button>` can never contain another `<button>`; interactive content can't nest. The browser's parser doesn't error on this, it silently *rewrites* the DOM: the instant it hits the first nested `<button>`, it force-closes the outer one early. This was verified concretely, not inferred from the spec alone тАФ headless Chrome's `--dump-dom` output on a test render showed `</div></button><button class="cn-auth-privacy">...`, meaning `.cn-auth-privacy`/`.cn-auth-signout` were real DOM **siblings** of `.cn-avatar`, not descendants of `.cn-auth-dropdown` at all, despite the JSX clearly nesting them there. This had apparently been true since the avatar dropdown was first built тАФ it just hadn't been visually obvious until this session's changes (a bigger identity block) made the resulting broken layout noticeable.

Fixed structurally: the trigger button and its dropdown are now **siblings** under one shared `position: relative` wrapper (`.cn-avatar-wrap`), never parent/child тАФ the same shape every other popover in this app already correctly uses (see the `.card-menu` gotcha above; that note previously and wrongly claimed the avatar dropdown followed this same pattern тАФ it didn't, until now). The outside-click-dismiss `avatarRef` moved from the inner `<button>` to this new wrapper `<div>`, since `.contains()` needs to cover both the trigger and the now-sibling dropdown to correctly treat clicks inside either as "not outside."

**On top of both fixes, gave the identity block real profile-card presence:**
- Added `.cn-auth-avatar`, a 36px filled circle (`--accent-soft` background, `--accent` text) showing the same initial as the header's small trigger avatar тАФ but sized and colored with real weight, since this is the profile card's one moment to be the focal point rather than compete with the rest of the header row.
- `.cn-auth-signout` gained an `MdLogout` icon, matching `.cn-auth-privacy`'s `MdLock` тАФ previously Sign out had no icon at all, an inconsistency once Privacy already had one.
- Verified all of this by rendering the trigger + dropdown with the *exact* real JSX nesting (a `<button>` wrapping a `<div>` dropdown, in the old broken version) via a headless-Chrome DOM dump тАФ not just a visual screenshot тАФ since a screenshot alone wouldn't have revealed a DOM-structure bug like this one.

## Trust Card (report #2)

`view === 'trust'` renders `<TrustCard firstDocId={...} onDismiss={...}/>` тАФ "Your Doc is the real thing. SnipKeep is just how it got there," with a real link to the user's actual first Doc. Auto-shows exactly once (gated by `chrome.storage.sync.hasSeenTrustCard`), triggered the moment `docs.length` goes from 0 to тЙе1 (either detected on drawer mount for pre-existing users, or via the `'docs'` branch of the storage-change listener, ~1.1s after the doc is added while the drawer is open тАФ the delay lets `DocsTab`'s own "added" flash register first). `trustShownRef` guards against the mount check and the listener both firing; the listener re-reads `hasSeenTrustCard` fresh from storage rather than trusting React state, avoiding a stale-closure bug (the listener closure is created once, at mount, in a `useEffect([])`). Reachable again anytime via Privacy Ledger's "Why your archive is safe тЖТ" link. Dismissing ("Got it") always returns to `'main'`, not back to wherever it was opened from тАФ deliberately shallow, no back-stack.

## Link-Rot Insurance (report #3)

Best-effort: a dead source page shouldn't kill the тЖЧ Source link forever. After a real save succeeds, `ensureArchived(url)` runs **fire-and-forget** (`.catch(() => {})`, not awaited) тАФ never blocks the save or delays the toast; any failure (timeout, network error, archive.org declining) is silent.

- Keyed by the **source page URL**, in `chrome.storage.local.archivedUrls: Record<url, snapshotUrl>` тАФ several clips from one article share one snapshot, and re-clipping that page weeks later into a different doc won't re-request it (`ensureArchived` checks the map first, and again right before writing, to survive two concurrent saves).
- `archiveSnapshot`: `GET https://web.archive.org/save/<url>` тАФ free, no API key. `fetch` follows the redirect chain automatically, so `response.url` **is** the permanent snapshot address; no response parsing needed. 15s `AbortController` timeout.
- Requires the `https://web.archive.org/*` host permission (**needs a full reload**, same as any permission change).
- **Known gap:** if the MV3 service worker unloads before the fetch resolves, that one snapshot is silently skipped. Acceptable тАФ it's a soft fallback, not core functionality, and the next clip retries for any still-unarchived page.
- UI: History's `archivedUrls` state loads once and stays live via a `storage.onChanged` listener, so the link can appear a few seconds after a save without reopening the drawer. `clipCard` shows **ЁЯПЫ Archived** only when a snapshot exists for that entry's `sourceUrl`.
- Once a snapshot exists, `ensureArchived` also writes it **into the Doc itself** (see Doc bookmarks below) тАФ the local `archivedUrls`/ЁЯПЫ-link UI is a convenience, not the source of truth; the Doc stays the durable copy, consistent with the Trust Card premise.

### Doc bookmarks (`NamedRange`) тАФ shared primitive behind link-rot write-back and Living Resurface

A `NamedRange` marks a position in a Google Doc that **Google keeps in sync automatically** as the surrounding document is edited тАФ the reliable "remember an exact spot, come back and edit near it later" building block. Created via a `createNamedRange` request folded into the *same* `batchUpdate` that writes the clip (no extra round trip); its `namedRangeId` comes back in that response's `replies[]` array, **parallel to `requests[]`** тАФ `appendToGoogleDoc`/`appendImageToGoogleDoc` track the index they pushed each `createNamedRange` request at (`bookmarkIdx` / `captionRangeIdx`) to pull the right reply back out. Both functions **return** `{ clipNamedRangeId, captionNamedRangeId }` instead of `void`.

Two bookmarks get created:
- **Caption bookmark** тАФ over `[captionStart, captionEnd-1]` ("domain ┬╖ date"), created on every new-article save (text or image). Persisted in `chrome.storage.local.docCaptionBookmarks: Record<url, Record<destinationId, namedRangeId>>` (`saveCaptionBookmark`) тАФ a page clipped into two different Docs gets an entry for each.
- **Clip-block bookmark** тАФ over `[clipStart, end]`, where `end` is `note ? noteEnd-1 : clipEnd-1` (the whole block: bullet **plus its margin note, if one exists at save time** тАФ so a later addition lands after both, not wedged in the middle). Created on **every** text-clip save. Stored directly on that `HistoryEntry.namedRangeId` тАФ no separate lookup table. **Text clips only**; images don't get one.

`resolveNamedRange(docId, token, namedRangeId)` тАФ `GET .../documents/{id}?fields=namedRanges` (partial response, cheap), linear-searches every name-group for the matching ID, returns the **last** sub-range's `endIndex`. Returns `null` if not found (e.g. the user deleted that part of the doc) тАФ every caller treats that as a normal, silent "skip," except the one interactive path below.

**Two very different failure philosophies on top of the same primitive:**
- `appendArchiveLinkToDoc` (from `ensureArchived`) тАФ **silent best-effort**: `getAuthTokenSilent()` (no prompt), try/catch, any failure just means no archive link this time.
- `handleAddDocNote` (behind `ADD_DOC_NOTE`, "+ Add a note" тЖТ "Add to Doc") тАФ **interactive and real errors on purpose**. Uses `getAuthToken()` (may prompt) and throws a genuine `"Couldn't find that clip in the Doc anymore"` if the bookmark resolves to nothing тАФ a deliberate user action, so silent failure would be wrong here. On success it patches the matching local `clips` entry (`savedAt === entrySavedAt`) so the History card reflects it immediately.

The dated note text (`(MMM D) your thought`) reuses `noteStyleRequests`, appended to the entry's stored `note` joined by `\n` тАФ `.history-note` has `white-space: pre-line` so that renders as real line breaks.

## Living Resurface (report #8, pulled forward)

"+ Add a note" is available on **any clip with a Doc bookmark**, not just the Resurfaced spotlight тАФ the write-back mechanism never cared which card triggered it, so restricting it to Resurfaced was a UI choice, not a technical one, and got lifted. In `Popup.tsx`, which card's note box is open is tracked by `noteOpenFor: number | null` (the clip's `savedAt`, a stable identity) тАФ **not** by the render `key` string, since the same underlying clip can appear twice on screen (once as the Resurfaced spotlight, once again in the regular list below) and both instances should share one open/closed state. See "Doc bookmarks" above for the write mechanism.

## Soft Triage (report #4)

`HistoryEntry.someday` is pure local archive metadata (`toggleSomeday`, no Docs call, no bookmark needed тАФ works on every clip regardless of age, unlike the bookmark-gated features above). Someday clips are hidden from the main list **by default**; the header's `ЁЯХТ Someday (N)` toggle (`showSomedayOnly`) is the one-click way back to them тАФ this is the actual point (fewer things in view), not just an optional filter nobody remembers to use.

`pickResurfaced` excludes someday clips (already triaged once) and done-project clips (see Assignment/Project Mode). `pickTriageCandidate(clips, excludeSavedAt)` occasionally surfaces a `triageCard` тАФ pool is not-someday, not today's Resurfaced pick, saved >14 days ago (`TRIAGE_MIN_AGE_MS`), needs тЙе3 candidates, same daily-deterministic-pick shape as `pickResurfaced` but offset (`+7`) so the two rarely coincide. All three of its actions тАФ *Yes, still relevant* / *Mark as Someday* / *Not now* тАФ do nothing more than record `triageDismissedDay` (a day-seed in `chrome.storage.local`) so it won't ask again today; **skipping has zero consequence**, deliberately the opposite of Burn 451's delete-if-unread mechanic. Styled calm on purpose тАФ dashed border, no accent color тАФ distinct from Resurfaced's violet "delight" treatment.

## Gentle Reflection Nudge (report #5)

`pickReflectionNudge` walks the newest-first archive from the front, counting a streak of consecutive clips sharing the same `sourceUrl` with no `.note` (breaks on the first clip from a different page or with a note); fires at `REFLECTION_NUDGE_THRESHOLD = 5`. Pure local computation, same shape as Soft Triage тАФ no background call, no new `HistoryEntry` field.

Dismissal (`reflectionNudgeDismissed: {url, count}`) is keyed by **streak length**, not just the URL: dismissing at 5 suppresses it, but if that same article's streak keeps growing to 8, it resurfaces тАФ the pattern intensifying is worth a second nudge, an unchanged streak isn't. *"Look back"* sets the search box to the article's title (reuses the existing substring search); *"Dismiss"* records the dismissal тАФ ignoring the nudge entirely does the same thing. Rendered between Resurfaced and Triage, and deliberately the lightest-weight of the three: no card box, no border, just an inline line.

## Deadline-Aware Citations (report #6)

`DocDestination.dueDate` (ISO `YYYY-MM-DD`) is local metadata like Someday, never written into the Doc. `deadlineStatus(dueDate, uncited)` computes `daysLeft` treating the due date as end-of-day (`T23:59:59`), so "due today" doesn't misread as overdue in the morning, and escalates **one status line** through three tiers тАФ calm (>7d) / warn (3-7d) / danger (тЙд2d, today, or overdue) тАФ rather than adding new elements as urgency increases.

`HistoryEntry.cited` is a one-way ratchet set by `handleCite` on a successful copy (never unset by re-clicking); the Cite button becomes **"тЬУ Cited"** once true, mirroring Someday's active-toggle look. `DocsTab` loads `clips`/`history` (in addition to `docStats`) to compute `uncitedCounts: Record<destinationId, count>`, kept live via the same `storage.onChanged` listener extended to watch `changes.clips` too.

The danger tier's **"Cite them тЖТ"** is a `<span onClick stopPropagation>` nested inside the status `<button>` (stopPropagation so it doesn't also fire the button's own "edit this deadline" handler) that calls `onJumpToHistory(doc.name)` тАФ lifted through the `Popup()` root as `historyFilter` state, since `DocsTab` and `History` are siblings with no direct line to each other. `History` receives it as `initialFilter`, drops it into its own search box via a `useEffect`, then calls `onFilterConsumed()` to clear the shared state (so revisiting History later doesn't reapply a stale jump).

New semantic color token `--warn` (amber) sits alongside the existing `--danger` for the escalation ramp тАФ kept separate from the violet `--accent`, per the design system's "accent reserved for ~4 tiny spots" rule.

### Custom calendar (`DeadlineCalendar`)

The date picker is a **custom calendar**, not a native `<input type="date">` тАФ the native picker is OS-rendered and can't be themed, so it would clash with the rest of the drawer. `toISODate(d)` formats using **local** year/month/day components (deliberately not `toISOString()`, which reads UTC and can silently shift the date by a day depending on timezone) тАФ this has to agree with `deadlineStatus`'s local-"today" comparison. Commits immediately on click (no separate Set step); dates before today are disabled; dismiss-on-outside-click via `composedPath()` + a ref, same pattern as the Drawer's avatar dropdown and the Toolbar's destination menu. `.cal-popup` is `position: absolute` anchored to `.deadline-row` (`position: relative`), floating over the list rather than pushing later cards down. Visually verified against the real `popup.css` at 340px before shipping тАФ grid alignment, selected/today color treatment, and the floating shadow all confirmed via a headless-Chrome screenshot.

## Assignment/Project Mode (report #7)

`DocDestination.done` splits destinations across two sibling tabs: `DocsTab` shows only `docs.filter(d => !d.done)` (a doc vanishes from Docs the moment it's marked done), and `CompletedTab` shows only `docs.filter(d => d.done)`.

**This started as a collapsed section inside DocsTab** (matching History's Someday-filter pattern) **but was deliberately promoted to a third top-level tab** at the user's explicit request, after a critique was given and heard (a third tab is a permanent cost for rarely-visited content, and breaks the just-established Someday precedent тАФ Hick's Law, tab-bar width risk at 340px, "Docs" would need a less-accurate rename). The user weighed it and chose the tab anyway. Visually verified at the real 340px width: three tabs fit with room to spare.

`DocsTab`'s `markDone(id)` (one-directional тАФ a doc marked done immediately leaves DocsTab's own list, so there's no "un-mark" path from here) sets `done: true` and shows a flash ("Marked as done тАФ see the Completed tab.") rather than auto-switching tabs тАФ confirms where the item went without yanking the user away from whatever else they were doing in Docs. `CompletedTab` is a **fully independent sibling** тАФ it can't reach into `DocsTab`'s state, so it loads its own `docs`/`docStats` and has its own `storage.onChanged` listener; safe because only one tab is ever mounted at a time. Its "тЖй Reopen" sets `done: false` (flash: "Reopened тАФ see the Docs tab."); "тЬХ" removes the destination entirely.

**Card simplification (same session, follow-up):** the active doc card originally had two separate icon buttons (тЬУ mark done, тЬХ remove) sitting next to the toggle switch. Consolidated into a single **`┬╖┬╖┬╖` menu** (`DocMenu` component) containing both actions, so the card's button row stays down to just the toggle + one trigger тАФ same floating-dropdown-with-outside-click-dismiss pattern as `DeadlineCalendar` and the Drawer's avatar menu. `menuOpenFor: string | null` tracks which card's menu is open (same shape as `editingDeadlineFor`). Visually verified against the real CSS: closed state is a clean two-control row, open state floats a small menu below the `┬╖┬╖┬╖` trigger without clipping.

**"+ Set a deadline" also moved into this menu (follow-up to the follow-up)** тАФ but only the *prompt* for adding one, not the deadline itself once it exists. `DocMenu` takes an optional `onSetDeadline`, passed only when `!doc.dueDate`; once a deadline is set, editing goes through the always-visible countdown pill (`onClick={() => setEditingDeadlineFor(doc.id)}`), not the menu тАФ burying an active "danger" countdown behind a menu click would undercut the entire reason Deadline-Aware Citations exists (ambient, can't-miss urgency). The `.deadline-row` wrapper itself is now conditionally rendered (`isEditingDeadline || status`) rather than always present with an empty/link fallback тАФ a doc with no deadline shows nothing extra at all now, no dangling row.

**Remove now asks first тАФ inline, not a modal.** Requested as "show a warning modal"; countered and the user picked the alternative (design + psychology critique, per the standing practice тАФ see the `critique-ux-ideas-first` memory): a real confirmation was warranted (removing silently drops the doc's deadline metadata, and "Remove" reads ambiguously close to "delete my Google Doc," which it never does), but a blocking modal-with-backdrop would've been the *first* one anywhere in SnipKeep тАФ the whole drawer is deliberately `modal={false}`, page-stays-interactive, precisely because this is meant to be a companion, not something that takes over the screen. Built as an **inline swap** instead: `DocMenu` holds a local `confirming` boolean; clicking "тЬХ Remove" swaps the dropdown's own content (not a new surface) to a short reassurance тАФ "Remove **sandeep2**? Your Doc and its clips stay safe тАФ this only stops saving here." тАФ with Cancel / Remove. The copy is doing real work: it answers the one thing someone would actually worry about at that exact moment, the same trust-reinforcement instinct behind the Trust Card and Privacy Ledger. `onRemove` only fires from the confirm click; Cancel drops back to the normal 3-item menu (not a full close) so a misclick doesn't cost an extra reopen.

**Readability pass (immediate follow-up).** The first version packed the question and the reassurance into one 3-line run-on sentence at 216px/12px padding тАФ cramped, and the reassurance (the actually-important part, psychologically) had no visual distinction from the rest. Before touching anything, the actual WCAG contrast was computed rather than assumed: `--text-2` on `--surface` is **7.53:1**, `--danger` on `--surface` is **6.6:1** тАФ both already comfortably past AA's 4.5:1, so contrast was never the real defect. The fix was hierarchy and room: split into `card-menu-confirm-title` ("Remove **sandeep2**?" тАФ 14px bold, `--text`, own line) and `card-menu-confirm-text` (the reassurance тАФ 12.5px, `--text-2`, own line) тАФ two tiers instead of one paragraph, so the reassurance can be scanned independently rather than parsed out of a sentence (chunking, Miller's Law). Card widened 216px тЖТ 240px and padding 12px тЖТ 16px so the reassurance wraps across two even lines instead of three cramped ones. Cancel/Remove buttons grew from `padding: 6px 11px` (borderline ~26px tall) to `8px 14px` + explicit `min-height: 32px`, comfortably past WCAG 2.5.8's 24px target-size floor rather than sitting right at the edge of it.

**Menu item alignment/spacing pass (same session).** The 3-item list (Set a deadline / Mark as done / Remove) rendered each item as a single inline string ("+ Set a deadline"), so the icon glyphs (`+`/`тЬУ`/`тЬХ` тАФ three very different natural widths) left every label starting at a slightly different x position. Fixed with a `.card-menu-icon` fixed-width (14px) flex column per item, so all three labels now line up on the same left edge regardless of which glyph precedes them. Item padding grew `7px 9px` тЖТ `8px 9px` with an explicit `gap: 1px` between items at the `.card-menu` container level (previously relying on padding alone with no defined rhythm between rows) for a clearer, more consistent vertical cadence. A `.card-menu-divider` now separates **Remove** from the two safe/reversible actions above it тАФ the same "give the destructive item distance" convention used in macOS/VS Code context menus, so a quick double-click right after Mark as done doesn't land on Remove by muscle memory (error prevention, not just decoration). (Classes were `.doc-menu*` at the time; renamed to `.card-menu*` once the History card started reusing them тАФ see below.)

**Stacking-context overlap bug (same session, found via screenshot report).** An inactive card's "┬╖┬╖┬╖" menu, opened near the bottom of the list, could render *behind* a later inactive card and be partly unreadable тАФ despite `.card-menu` having an explicit `z-index: 20`. Root cause: `.doc-item.inactive` (and `.doc-item.done`) sets `opacity: 0.5`, and **any element with `opacity < 1` establishes its own CSS stacking context** тАФ so when the card whose menu is open is itself dimmed, the menu (however high its own z-index) is trapped inside that card's stacking slot. Sibling-vs-sibling paint order is then decided one level up, by the *cards'* stacking positions, not the menu's тАФ and a later sibling that's *also* opacity-dimmed competes at that same level and wins by DOM order, painting over the earlier card's entire subtree, menu included. (This is also why the menu still appeared *in front of* the non-dimmed "Add document" button below it тАФ plain undimmed elements don't get this implicit stacking promotion, so they lose to the dimmed card's context normally.) Fixed by promoting the *card itself*, not the menu: while a card's popover (menu or deadline calendar тАФ `menuOpenFor === doc.id || isEditingDeadline`) is open, it gets a `.popover-active` class (`position: relative; z-index: 30`), which lets that whole card's stacking slot outrank any opacity-dimmed sibling regardless of DOM order. Checked the History tab's card menus for the same trap тАФ they're already safe, since `CiteMenu`/`HistoryCardMenu` render as siblings of the opacity-animated `.history-actions-row`, not descendants of it (a structural choice originally made to dodge a *different* clipping bug, which happens to dodge this one too).

**Active docs sort above inactive, with a FLIP slide (same session).** `activeDocs` is sorted `active`-first (stable, so order within each group is preserved) тАФ usable toolbar destinations stay on top, toggled-off ones sink to the bottom. Toggling a doc therefore *moves* its card, and to avoid a disorienting teleport it animates via **FLIP** (First-Last-Invert-Play, no library): a `useLayoutEffect` iterates `activeDocs` in its current visual order each render (not `Map` insertion order тАФ needed for the stagger below) and records each card's `offsetTop` (layout-relative, not `getBoundingClientRect` тАФ immune to scroll between renders, ignores in-flight transforms). When a card's top changed, its delta plays as a slide-into-place. Honors `prefers-reduced-motion` (snaps, no slide).

**Active/inactive boundary divider (same session, follow-up).** The sort above groups active-before-inactive, but a sort alone is invisible тАФ nothing signaled *why* cards started dimming partway down the list, so it could read as arbitrary rather than a deliberate grouping. Fixed with a Gestalt "common region" cue: `showInactiveDivider = i > 0 && activeDocs[i-1].active && !doc.active` detects the exact transition point while mapping, and renders a `.doc-list-divider` there тАФ a small uppercase label (`.section-label`'s established typographic treatment, reused rather than inventing a new one) plus a hairline rule filling the remaining width. The label reads **"Hidden from toolbar"**, deliberately matching the toggle's own tooltip text (`'Hide from toolbar' : 'Show in toolbar'`) rather than introducing new vocabulary for the same concept. Spacing is asymmetric on purpose: `margin-top: var(--space-2)` on top of the list's own flex `gap` doubles the space above the divider (16px vs. the normal 8px between cards) while the gap below stays standard тАФ reads as "attached to what follows, broken from what came before," the usual convention for a group label. Implemented via a `React.Fragment key={doc.id}` per iteration (needed to render an optional sibling before the keyed card without changing the map's key structure) тАФ doesn't interfere with the FLIP measurement above, since that only ever reads each `.doc-item`'s own `offsetTop`, which already reflects however much space the divider adds or removes above it.

**"Add document" repositioned to right after the active group (same session, follow-up).** It used to sit unconditionally at the very bottom of the tab, after the whole list тАФ which, once the divider above existed, put it directly underneath "Hidden from toolbar," an odd adjacency (adding a destination is an active-oriented action; a fresh doc is active by default) and one that gets harder to reach as more docs get toggled off over time. `lastActiveIdx = activeDocs.reduce((acc, d, idx) => d.active ? idx : acc, -1)` finds the last active doc's index once per render; the shared `addControl` (the same trigger-button-or-expanded-form JSX, extracted to a variable so it isn't duplicated) is inserted right after that card, before the divider. Two fallbacks handle the cases where there's no "after the active group" position to hook onto: if **no** doc is active (`lastActiveIdx === -1`), it renders at the top of the list instead of vanishing; if the list doesn't render at all (`activeDocs.length === 0` тАФ genuinely zero docs, or every doc marked done), it renders below the list as before. Stays inside the same single `.doc-list` container as the cards and divider (not a second list) тАФ critical for the FLIP animation, which requires each `.doc-item` to remain the same DOM node across renders; splitting active/inactive into two separate list containers would force React to unmount and remount a card the moment it crosses the boundary, destroying the ref FLIP depends on.

**Two implementation passes тАФ the first had two real bugs, not just "wrong numbers."** The initial version hand-toggled CSS: set an instant inverting `el.style.transform`, forced a reflow, then released it under `el.style.transition` inside a `requestAnimationFrame`, cleaning up on `transitionend`. Two problems surfaced after "still not smooth" feedback:
1. `el.style.transition = 'transform ...'` **replaces** the card's own CSS transition for its hover highlight (`background`/`box-shadow`). Since the mouse sits on the card when its toggle is clicked, that hover state is live during the slide тАФ so the highlight was snapping instead of fading throughout the animation, reading as jank even when the slide's own motion was fine.
2. The easing tried at that point, Material 3's "emphasized decelerate" (`cubic-bezier(0.05, 0.7, 0.1, 1)`), is **not actually gentle at the start** тАФ its first control point `(0.05, 0.7)` reaches 70% speed after only 5% of the distance. That curve is built for elements already in motion entering the screen; used on something starting completely at rest, it blasts off almost instantly. "Gentle" requires a genuine zero-velocity start, which this curve doesn't have despite its soft landing.

**Fixed by switching to the native Web Animations API** (`el.animate([...], { duration, delay, easing, fill: 'none' })`) instead of manual style toggling. This runs as an independent animation layer that can never touch or conflict with the element's CSS transitions (fixes #1), and removes the manual reflow-forcing/`requestAnimationFrame`/`transitionend`-cleanup choreography entirely тАФ the browser applies the first keyframe deterministically, no timing tricks needed. Easing switched to Material's actual **"standard" curve**, `cubic-bezier(0.4, 0, 0.2, 1)` тАФ its `(0.4, 0)` control point gives a real ease-in from zero velocity (fixes #2).

Two more levers beyond raw duration, both still in place: **distance-proportional duration** (`Math.min(700, 380 + |dy| * 0.4)` ms тАФ a fixed duration for every distance breaks the naive-physics expectation that farther travel takes longer) and a **per-card stagger** (`delay = Math.min(150, index * 20)` ms, keyed by the card's position in the new order тАФ when several cards move at once, perfectly synchronized identical motion reads as mechanical; real independent objects never move in exact lockstep, so a small top-to-bottom offset turns a rigid block-move into a cascade).

**Third pass тАФ actually instrumented and measured, not re-guessed (same session).** After "still not smooth" feedback a third time, built a standalone harness (real `popup.css`/font, a verbatim port of this exact JS logic) and drove it via live browser automation instead of reasoning blind again. That surfaced a real methodology limit: the automated tab ran with `document.visibilityState === 'hidden'`, which throttles `requestAnimationFrame`/`setInterval` heavily and made raw in-browser timing samples intermittently unreliable (some captures silently stalled for seconds). One clean sample set was captured before that (delay=0 card, `cubic-bezier(0.4, 0, 0.2, 1)`, 18 monotonically-decaying data points) тАФ and it, plus a deterministic bezier solve (pure math, immune to any browser/tab quirk) done independently to cross-check it, agreed: that curve reaches **~24% progress by 25% of its duration** тАФ close to linear, not gentle, despite a technically-zero tangent at t=0. Switched to `cubic-bezier(0.65, 0, 0.35, 1)` (a pronounced, genuinely symmetric ease-in-out): only **~7% progress by 25% of duration**, mirrored at the tail (~93% by 75%) тАФ a real slow-in *and* slow-out. Duration cap trimmed `700ms тЖТ 650ms` to keep repeated toggling from feeling sluggish. This also retroactively explains the very first "very sudden" report: that pass used a CSS-transition-based mechanism (since replaced by `el.animate()`) that also happened to clobber the card's hover-highlight transition тАФ so the earliest complaint likely had two compounding causes, not one.

**Still wants a live click-test in the loaded extension** for final confirmation тАФ the automation testing above validated the curve mathematically and the mechanism structurally, but real perceived feel (and whether 650ms reads as gentle vs. sluggish in practice) is the one thing only an actual focused-tab, human-eyes check can settle.

**Tab-switch entrance animation removed (later session).** `.tab-content` briefly carried its own mount animation (`cn-tab-in` тАФ fade + 6px settle, fired on every tab switch since each tab is a distinct component that unmounts/mounts). Removed by request: the deliberate, carefully-tuned motion in this popup is the card-reorder slide above, not a whole-section fade on every tab click тАФ the two didn't conflict mechanically, but a second, separate motion at the tab level was more than needed. Tab switching is now an instant swap; only `DocsTab`'s card reorder animates.

**The actual "toggle isn't smooth" bug тАФ a different element entirely (later session).** After repeated "still not smooth" reports on toggling a doc, all previous passes had tuned the *card-reorder slide* тАФ the toggle switch's own knob had never once been examined. It had a real, separate bug: `.btn-toggle::after` animated the CSS `left` property (`transition: left 0.2s`, no explicit easing). `left` is a **layout property** тАФ animating it forces the browser to recompute layout and paint on every frame, a well-known cause of janky, non-compositor-smooth motion, worse inside a content script's Shadow DOM competing with the host page's own rendering. Fixed by switching the knob to `transform: translateX(...)` (compositor-only, no layout recalculation) with an explicit `cubic-bezier(0.4, 0, 0.2, 1)` easing instead of the implicit default `ease`. `left: 2px` is now a fixed base position; the two states (`on`/`off`) differ only by `translateX` distance (`10px` / `0`). No other element in the codebase had the same `left`/`top`-animation anti-pattern (checked). This is very possibly what several rounds of card-animation retuning were actually missing тАФ the complaint may have been about the switch itself the whole time, not the card slide.

**The "sudden jerk" on toggle тАФ a THIRD, independent bug, in the card itself (same later session).** After the knob fix, the user clarified the jerk was specifically in the *document card's* vertical movement, not the knob. Found: `.doc-item`'s CSS `transition` covers only `background` and `box-shadow` тАФ **never `opacity`** тАФ yet `.doc-item.inactive` sets `opacity: 0.5`. So the instant a card toggles inactive, its opacity snapped from `1 тЖТ 0.5` with **zero transition**, at the exact same instant the FLIP slide began. A sharp, un-animated visual change (instant dimming) landing at the exact start of otherwise-smooth motion is precisely what reads as a "jerk," regardless of how good the sliding itself is тАФ the two just needed to happen *together*, gradually.

Fixed by folding the opacity change into the **same** `el.animate()` call as the position slide, rather than adding a separate CSS `transition: opacity`. This distinction matters: a separate CSS transition (fixed duration) could never stay perfectly in sync with the position slide's duration, which is dynamically computed from distance (`460тАУ780ms`) and per-card stagger тАФ two independently-timed transitions risk drifting out of step with each other, which would just trade one kind of visual mismatch for another. Driving both properties through one `animate()` call with one shared `duration`/`delay`/`easing` guarantees they're always in lockstep.

Implementation: a new `prevActiveMap` ref (parallel to `prevTops`) records each card's active state as of the last render. `fromOpacity`/`toOpacity` are derived from the previous vs. current active state (`1` active / `0.5` inactive, matching the CSS values exactly, so there's no flash when `fill: 'none'` hands control back to the CSS rule at the end). The `el.animate()` keyframes now carry both `transform` and `opacity` together:
```js
el.animate(
  [
    { transform: `translateY(${dy}px)`, opacity: fromOpacity },
    { transform: 'translateY(0)', opacity: toOpacity },
  ],
  { duration, delay, easing: 'cubic-bezier(0.65, 0, 0.35, 1)', fill: 'none' }
)
```
The effect's trigger condition also changed from "did position change" alone to "did position **or** active-state change" (`positionChanged || opacityChanged`), so a toggle that doesn't visually move a card (e.g., no reordering needed) still gets a smooth opacity fade instead of an instant snap.

`done` is **deliberately independent of `active`** тАФ marking a project done doesn't silently hide it from the toolbar, and reopening doesn't silently restore that either; both stay under the user's explicit control. `History` also loads `docs` to build `doneDestIds: Set<string>`, kept live via its own `storage.onChanged` handler; before calling `pickResurfaced`/`pickTriageCandidate`/`pickReflectionNudge`, entries are narrowed to `activeEntries` (excludes clips whose `destinationId` is done). Only the **proactive pickers** are affected тАФ the main searchable History list is untouched, so a finished project's clips stay fully findable and citable, they just stop being surfaced unprompted.

## Topic Auto-Clustering (report #10) тАФ built, then removed

Shipped as a row of tag/domain pill chips above History's search box (each click dropped its label into the search query), then **fully removed** 2026-07-06 after a UX critique тАФ deleted, not paused (`pickTopicClusters`, `extractTags`, `TopicCluster`, and the `.topic-*` CSS are all gone). Two reasons it didn't earn its place:

1. **No signal at realistic scale.** The chips showed unconditionally, so a domain that covered ~100% of a small archive produced a chip that filtered to "everything," and a just-applied tag produced a chip revealing nothing. That's a progressive-disclosure failure тАФ complexity shown when it has nothing to say.
2. **Duplicated an existing affordance.** The recognition/browse value it targeted (recognition-over-recall vs. the blank search box) was already delivered inline: every `#tag` on a card is a clickable filter (`renderNoteWithTags`) and each card shows its source domain. The chips mostly re-surfaced what the cards already exposed; their only unique value (aggregate counts + off-screen clusters) only matters in a large, varied archive.

**One keeper from that work:** History's search predicate still includes `e.sourceUrl.toLowerCase().includes(q)` тАФ originally added so domain chips would match, now retained as an independently useful capability (typing a bare domain like `nytimes.com` filters History by site). If clustering is ever revisited, it should appear only at real scale behind a hard "the cluster must actually discriminate" gate (e.g. exclude any cluster covering most of the archive), never shown unconditionally.

## Filter History by document

A **Filter** control in the History header narrows the clip list to a single destination doc. Deliberately *not* the same thing as searching a doc name тАФ it's **recognition** (a `FilterMenu` dropdown lists the docs you actually have clips in, each with a count) rather than recall (typing a name), and it matches by exact `destinationId`, not a name substring that could also hit clip text. `docFilter: string | null` holds the selected id; the filter composes with search (`effectiveDocFilter && e.destinationId !== effectiveDocFilter` short-circuits before the search predicate) and with the Someday split.

- **Progressive disclosure:** the whole control only renders when `docOptions.length > 1` (`canFilterByDoc`) тАФ a filter that can only pick the single existing doc is pointless.
- **Self-healing:** `effectiveDocFilter` re-validates against the current doc list each render, so if docs drop to тЙд1 or the filtered doc loses all its clips, the filter auto-disables rather than stranding the view on an empty list with no visible way to clear.
- **Status without crowding:** the header trigger stays a fixed-width "Filter" pill (accent when active). *Which* doc is active shows as a separate removable **accent chip below the search** (`filter-active-chip`, "тЫГ sandeep2 тЬХ", whole chip clears) тАФ putting the doc name on the trigger itself overflowed the 340px header once Someday + Clear all were also present. The header (`.history-header`) also `flex-wrap`s so the count stays intact and the action pills drop to a second line, right-aligned, in the busiest state instead of the count breaking mid-word.
- `FilterMenu` reuses the shared `.card-menu` surface (with a "All documents" reset row + divider), same family as the Cite / "┬╖┬╖┬╖" menus. It lives in the header (no overflow-clipped ancestor there), so unlike the card menus it needs no special anti-clip structuring.

## Deleting clips (single, bulk, and "Clear")

There was no way to delete one clip тАФ only the blunt "Clear all." Filled that in, and used the occasion to fix where the destructive control lives.

**Single-clip "Remove from history"** sits at the bottom of a card's "┬╖┬╖┬╖" menu, divider-separated and danger-styled (distance from the safe actions, same convention as the Doc menu's Remove). Labelled *history*, not "Delete", on purpose: it only drops SnipKeep's local record; the clip's text stays in the Google Doc (SnipKeep never destructively edits the Doc тАФ the whole trust premise). No blocking dialog тАФ an **undo bar** is the safety net instead, which beats a confirm for a frequent, low-stakes, reversible action.

**"Clear" was moved out of the header, to the bottom of the list** тАФ a deliberate response to a position critique. A destructive + rare action doesn't belong in the prime top-right zone next to the *safe* filter controls: it's in the accidental-tap path (adjacent to Filter), carries the same visual weight as benign controls, and proximity miscategorizes it as "another view control." At the bottom it's out of that path (reached by a deliberate scroll) and semantically "you've scrolled past everything тЖТ clear it."

**"Clear" acts on the current view** (one mental model: *clear what I'm looking at*). `isNarrowed = q || effectiveDocFilter || showSomedayOnly`: when narrowed it's **"Clear these N"** and clears just `filtered` (this is the recommended bulk-delete тАФ express *which* clips via the filter you already have, no checkbox multi-select mode); otherwise **"Clear all history"** clears every entry (including hidden Someday clips). Given the higher stakes it gets an inline confirm (`clearConfirming`) *and* the undo bar.

**Undo mechanism.** All removal paths go through `commitClips(next)` (writes state + `chrome.storage.local.clips` together; when the archive empties it also drops the legacy `history` seed, or the loader would resurrect the old last-10 on remount) and `stageUndo(snapshot, label)`, which keeps the whole pre-delete array for 6s so one restore path (`applyUndo`) covers single and bulk alike. The `.undo-bar` is `position: sticky; bottom` so it stays visible whether the delete happened at the top (a card) or the bottom (clear). Multi-select checkbox mode was explicitly **rejected** (see the design discussion) тАФ it introduces a mode for a rare action and leans into the manual-curation behavior the product is built to discourage; "filter + Clear these N" + single-remove covers the same intents without a mode.

**Note:** the sticky undo bar's exact scroll behavior is the one thing that wants a live click-test in the loaded extension (static/headless can't exercise `position: sticky` against the drawer's real scroll container); its non-sticky fallback (a normal bar at the list bottom) is also acceptable.

**Structural note тАФ actions row vs. its menus (important, easy to re-break).** To keep resting cards compact, the button row reveals on hover/focus by animating **`grid-template-rows: 0fr тЖТ 1fr`** on `.history-actions-row`, with the buttons in an `overflow: hidden; min-height: 0` inner (`.history-actions-inner`). Grid-rows is used deliberately instead of `max-height`: `1fr` resolves to the *exact* content height, so there's no overshoot gap to freeze on at the tail of the transition (that overshoot was a visible "jerk," worst on collapse). The gap to the meta lives as `padding-top` on the inner so it grows as part of the same animated height, not as a separate margin. The `overflow: hidden` on the inner **clips any absolutely-positioned popover opened from inside it** тАФ an earlier pass had the `┬╖┬╖┬╖` menu nested in the row and it rendered completely invisible (fully clipped). So the pop-open menus (`CiteMenu` and `HistoryCardMenu`) render as **siblings of the row**, direct children of the `position: relative` `.history-actions` wrapper тАФ outside the clip. The trigger buttons stay in the collapsing inner; the menus float free. If you ever move a menu back inside `.history-actions-row`/`-inner`, it will silently vanish. (This differs from the Doc card, whose menu can nest in `.card-menu-wrap` because the doc card has no overflow-clipped row.)

Removing the rename affordance doesn't just simplify the form; it makes the UI stop promising something the app was already quietly undoing.

Removed: `nameEditMode` state, `nameTouchedRef` (the guard that used to stop the auto-fetched title from clobbering a manually-typed name тАФ no longer needed since there's no manual name to protect), `handleNameChange`, and the `.name-preview`/`.name-preview-edit`/`.name-preview-skeleton` CSS.

Replaced with `.doc-name-status`, a plain read-only line between the URL field and the hint тАФ deliberately *not* styled like a field (no border, no input framing, no hover/click affordance) so it doesn't imply an editability that wouldn't stick anyway:
- Fetching: reuses the `.field-spinner` from the pass above + "Fetching the document's titleтАж"
- Resolved: an `MdCheckCircle` + `Will be added as "{title}"` тАФ a straightforward confirmation, not a control.

The name is now driven exclusively by the auto-fetched title (falling back to `'My Notes'` if the fetch fails or the doc is inaccessible), same as it always ultimately would settle to via `syncDocNames` тАФ just without the detour through a UI that implied otherwise.

**Follow-up: making sure a rename in the real Google Doc actually reaches SnipKeep.** With renaming removed from the UI, the user asked to confirm the reverse direction stays solid тАФ if someone renames the Doc itself (in Google Docs, not SnipKeep), SnipKeep should pick that up. `syncDocNames` already re-fetches every doc's live title, but it only ran once per `DocsTab` mount. That mount happens more often than it looks тАФ `Drawer.tsx` fully unmounts on close (not just hides), and `Popup.tsx` renders each tab via a ternary (`tab === 'docs' ? <DocsTab/> : ...`), so switching to Completed/History and back to Docs also remounts `DocsTab` and re-triggers the sync. Those two paths cover most real usage.

The one gap: the drawer staying open on the Docs tab the *entire* time, uninterrupted, while the Doc gets renamed in a different browser tab. Closed with a `document.visibilitychange` listener тАФ even though the drawer is a content-script overlay, it shares the host page's own `document`, so switching to another tab and back fires this exactly like it would on a normal page. A `docsRef` (kept current via a small `useEffect` mirroring `docs`) lets the listener always read the latest doc list without re-subscribing itself every time `docs` changes. No polling, no timers тАФ purely event-driven, matching this app's existing style.

Once `syncDocNames` writes the refreshed names to `chrome.storage.sync`, `CompletedTab` and `History` pick them up for free тАФ both already listen for `changes.docs` via their own `chrome.storage.onChanged` handlers (built for other reasons), so no additional wiring was needed for the name to reach every tab that displays it.

### Toolbar not appearing on Google Drive file previews тАФ the real fix needed two passes

First pass (previous entry, "only exclude the Docs editor") wasn't enough тАФ the user tried it, reloaded the extension and the tab, and the toolbar still didn't appear on a `docs.google.com/file/d/.../preview` page. Root cause was one layer deeper than the `exclude_matches` pattern.

**Google Drive's file preview renders its actual selectable content тАФ the PDF/text layer тАФ inside a nested `<iframe>`, not the top-level document.** Chrome content scripts only inject into a page's top frame unless the manifest says otherwise (`all_frames`, default `false`). So even with the exclusion narrowed correctly, the content script was still only running in the outer preview page's chrome (header, sidebar) тАФ never in the iframe where the actual document content, and therefore the actual text selection, happens. `mouseup` events inside that iframe were invisible to a listener registered only in the parent frame.

**Fix: `"all_frames": true` in `manifest.json`.** This makes the content script run in every frame of every matched page, so the toolbar's local `mouseup` detection now reaches into the iframe. But this single line has a real, non-obvious side effect covered in `CLAUDE.md`'s new gotcha: `chrome.tabs.sendMessage` without an explicit `frameId` broadcasts to *every* frame, not just the top one тАФ a fact confirmed against Chrome's own documentation, not assumed, since getting this wrong would have silently reintroduced a worse bug (duplicate drawers, spurious toasts) on any page with iframes, which is nearly every real website (ads, embeds, widgets). Fixed per-message: `TOGGLE_DRAWER`/`CAPTURE_IMAGE` now specify an explicit `frameId` from the background (`0` for the drawer, the context-menu click's own `frameId` for images); `TRIGGER_SAVE` can't be targeted that way (`chrome.commands` carries no frame info) so its content-script handler instead opens with a `document.hasFocus()` guard, letting only the actually-focused frame act on it.

This is the kind of fix that's easy to verify is *necessary* (the manifest change alone doesn't work, confirmed by the user re-testing) but hard to verify is *sufficient* without live browser testing across multiple frame scenarios тАФ flagged here for future sessions rather than treated as fully closed on code-reading alone.

### Follow-up: still didn't work for PDFs specifically тАФ because it architecturally can't

The user re-tested after the `all_frames` fix and the toolbar still didn't appear on the same file. Rather than guess a third time, this was investigated live with real browser automation (`mcp__claude-in-chrome__*`), navigating to the exact URL and inspecting the actual page.

**First finding, initially misread as "Google's viewer uses an opaque custom renderer":** live DOM inspection of both `docs.google.com/file/d/.../preview` and the fuller `/view` page found: 4 iframes total, none large enough to hold the visible content (the largest is 148├Ч148px against a full-page document); no `<canvas>`, no `<embed>`/`<object>`, no Shadow DOM, no text nodes matching the visible book text anywhere reachable; and тАФ the most direct test тАФ making a real click-drag selection produced a visible blue highlight on screen, but `window.getSelection()` came back completely empty, in the top document and in every same-origin iframe, checked immediately after the drag.

**The user then correctly pushed back: normal copy/paste actually works for this text.** That's real evidence a genuine selection mechanism exists тАФ it meant the first pass's explanation (implying nothing real was there) couldn't be the full story, and needed reconciling rather than defending. Re-investigated rather than restating the same conclusion:

**Resolving test: does the exact same thing happen on a plain PDF with zero Google involvement?** Navigated straight to a public PDF (`arxiv.org/pdf/...`). Chrome silently rewrote the URL to `chrome-extension://efaidnbmnnnibpcajpcglclefindmkaj/...` тАФ **Chrome's own built-in PDF Viewer** тАФ and the browser-automation tooling itself then refused to send input or execute JavaScript on that page, with the exact same error Chrome gives for internal pages like `chrome://settings`.

That reconciles every observation at once: Google Drive's preview very likely embeds this same privileged, native PDF-rendering surface internally. It handles select/copy/paste at the browser's own (C++) level тАФ genuinely, which is why copy/paste works тАФ without ever touching the standard web `Selection`/`clipboardData` APIs that any content script (SnipKeep's or any other extension's) can observe. Chromium documents this surface as deliberately walled off from content-script injection, the same restriction that applies to `chrome://` pages. Three independent confirmations: Chromium's own documented behavior, the automation tooling's own refusal to touch it, and the exact symptom pattern (real functionality, zero DOM footprint) this predicts for Google's embedded version.

**Conclusion: this is not a SnipKeep bug and not fixable by SnipKeep.** No manifest permission, no content-script technique, closes this gap тАФ Chrome deliberately isolates its PDF-rendering path from every extension, not just this one. The `all_frames`/`exclude_matches` fixes above remain correct and necessary for everything that *is* reachable (ordinary web pages, Sheets, Slides, Forms, any real DOM-based iframe) тАФ they were never going to reach PDF rendering specifically, anywhere in Chrome, and that's now confirmed rather than assumed.

## Voice-note capture (`docs/IDEAS.md` #1)

Hold the mic button in the Toolbar's note panel and speak a margin note instead of typing it тАФ transcribed live via the Web Speech API. Chosen from a fresh 5-idea creative brainstorm (`docs/IDEAS.md`) as the first to build, on the reasoning that a note-less clip is far more likely to become dead weight (the collector's fallacy this whole project is built around), and typing is the exact friction that makes people skip adding one.

**Why this needed an offscreen document, not a direct `SpeechRecognition` call in the Toolbar.** Researched before writing any code (see `CLAUDE.md`'s architecture section for the short version): a mic permission requested directly from a content script is scoped to *whatever website the user happens to be on* тАФ Chrome would prompt "nytimes.com wants to use your microphone," asked again on every new domain, and silently refused outright on sites that set a restrictive Permissions-Policy header. The documented, correct fix is `chrome.offscreen` (`reasons: ['USER_MEDIA']`) тАФ a hidden extension page with its own stable `chrome-extension://` origin, where the permission is asked once, ever, and works identically everywhere after.

**Getting it into the build at all was its own small discovery.** Offscreen documents aren't declared anywhere in `manifest.json`'s schema тАФ `chrome.offscreen.createDocument()` takes a bare URL string at runtime, nothing the manifest format has a slot for. `vite-plugin-web-extension` (this project's installed v4.x line) only bundles what's reachable from the manifest by default, so without `additionalInputs: ['src/offscreen/index.html']` in `vite.config.ts`, the file would silently never appear in `dist/` at all. Verified this precisely against the *installed* package's own `.d.ts` (`additionalInputs?: string[]`), not a docs page тАФ a websearch initially turned up a conflicting answer (an object shape with `scripts`/`html`/`styles` keys) that turned out to describe a different major version of the plugin.

**The message-routing design is the part most likely to bite a future change.** `chrome.tabs.sendMessage` with an explicit `frameId` targets exactly one frame; `chrome.runtime.sendMessage` тАФ the *only* option available to talk to an offscreen document, which isn't a tab and can't be reached via `chrome.tabs.*` тАФ broadcasts to every extension context at once (every open tab's content script, the offscreen doc, everything). Five message types exist specifically so no name is ever reused across two different hops, which would otherwise risk a listener seeing the same event delivered twice:
- `START_VOICE_NOTE` / `STOP_VOICE_NOTE` тАФ Toolbar тЖТ background.
- `OFFSCREEN_START_RECOGNITION` / `OFFSCREEN_STOP_RECOGNITION` тАФ background тЖТ offscreen doc, sent only after `ensureOffscreenDocument()` confirms the doc exists (creating it on first use).
- `VOICE_RECOGNITION_EVENT` тАФ offscreen doc тЖТ background. The background's listener defensively rejects any instance of this message that carries a `sender.tab` тАФ it should only ever arrive from the offscreen doc, which has none.
- `VOICE_NOTE_UPDATE` тАФ background тЖТ Toolbar, via `chrome.tabs.sendMessage` targeting the exact `{tabId, frameId}` captured when `START_VOICE_NOTE` was first handled. This pairing is tracked in a plain in-memory variable in the background (`voiceNoteTarget`) тАФ correct for this specific case, unlike the durable-storage pattern used everywhere else in this codebase, because a voice session is inherently live and short-lived: if the service worker is killed mid-recording, that one session simply stops, which is an acceptable degradation, not a correctness bug.

**Transcript merging happens in the Toolbar, not the offscreen doc.** The offscreen script has no notion of "what's already typed in the note field" тАФ it only ever streams the raw accumulated transcript for the *current* recording session. The Toolbar captures the note's existing value the instant recording starts (`baseNoteRef`) and, on every update, sets the note to `baseNote + (baseNote ? ' ' : '') + sessionText` тАФ replacing, never appending to, the session portion. This matters because `SpeechRecognition`'s `onresult` re-fires with the *cumulative* text of the current phrase each time, not just the newly-recognized fragment; naive appending would duplicate words on every interim update.

**Failure modes are explicit, not silent:**
- Unsupported browser тАФ feature-detected directly in the Toolbar itself (`'webkitSpeechRecognition' in window`), not via a round-trip to the offscreen doc. This is deliberate: an API's mere *existence* isn't origin-restricted, only actually invoking it (and touching the microphone) is тАФ so checking for it in the content script's own `window` is valid, and skips an unnecessary message hop. The mic button simply doesn't render at all when unsupported.
- Permission denied тАФ surfaces as inline text under the note panel (`.note-voice-error`), the recording state reverts to idle, and the button is immediately retriable (denial isn't remembered/blocked across attempts, in case the user changes their mind or fixes a site permission).
- Recognition ending on its own (silence, or the engine's internal timeout) is treated identically to an explicit stop click тАФ both just flip `isRecording` back to false.
- A 90-second safety timeout in the offscreen script force-stops a forgotten-open mic, since `continuous: true` would otherwise keep listening indefinitely with no natural end condition.
- If the Toolbar itself unmounts (save or dismiss) while still recording, a cleanup effect sends `STOP_VOICE_NOTE` so the offscreen doc doesn't keep listening with nothing left to receive its updates.

**Verification:** `tsc --noEmit` clean; build confirmed the offscreen document actually lands in `dist/src/offscreen/` (not just `src/content/` and `src/background/`) and that expected identifiers compiled into all three bundles. All three UI states (idle mic, recording/pulsing, permission-denied error text) were rendered against the *real*, extracted `Toolbar.tsx` `STYLES` string via headless Chrome, not a recreation. **Not yet live-tested** тАФ the actual microphone permission prompt, live transcription accuracy, and the stop/resume interaction all need a real Chrome session with a real microphone, which isn't reachable through headless automation; this is the one open item before calling the feature done, tracked in `tasks/todo.md`.
